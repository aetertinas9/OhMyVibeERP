# Redteam.md — 회차별 적대적 검증 (기록 전용)

매 회차 종료 시 그 회차의 구현을 적대적(red team) 관점으로 검증하고 결과를 기록한다.

## 용도 제한 (중요)

- 이 문서는 "바이브 코딩으로 엔터프라이즈 ERP를 만들 수 있는가" 실증을 위한
  **분석 전용 기록**이다.
- 여기 기록된 발견·지적 사항은 **이후 코딩 구현에 활용하지 않는다.**
  구현은 오직 사용자 지시(Vibe.md에 기록되는 프롬프트)로만 진행한다.
- 검증자는 발견 사항을 고치라고 제안하지 않는다. 판정과 근거만 남긴다.

## 검증 관점 (엔터프라이즈 ERP 기준)

1. 데이터 무결성 — 트랜잭션 경계, 동시성 제어, 정합성 제약, 이중 입력 방지
2. 보안 — 인증/인가, 역할·권한 분리, 입력 검증, 감사 추적(audit trail)
3. 업무 완결성 — 실제 업무 흐름의 끝단 처리(예외·취소·정정·마감), 법정 요건
4. 확장성·운영 — 데이터 규모 확장, 마이그레이션, 백업/복구, 장애·모니터링
5. 품질 — 테스트 존재 여부와 깊이, 에러 처리, 경계값·비정상 입력

## 회차 기록 템플릿

```
### Round NN (YYYY-MM-DD, branch: <브랜치명>, 기준 커밋: <해시>)
- 검증 방법: <어떤 공격 시나리오/시도를 했는지>
- 발견 사항:
  - [RT-NN-01] <관점 번호> <발견 내용과 재현 방법>
- 판정: 충족 / 부분 충족 / 미충족 — <근거 요약>
```

---

### Round 01 (2026-07-17, branch: main, 기준 커밋: 9158594)

- 검증 방법: 인증 경계의 소스 검토와 잘못된 자격 증명, CSRF 변조, 반복 실패,
  만료·폐기 세션, 과대 본문, 비인증 보호 페이지 접근 시나리오를 자동화 테스트로 실행했다.
- 발견 사항:
  - [RT-01-01] 2 보안 — 계정과 세션·시도 제한 상태가 단일 프로세스 메모리에만
    존재해 재시작 시 세션과 잠금이 모두 초기화되며 다중 인스턴스 간 상태가 일치하지 않는다.
  - [RT-01-02] 2 보안 — 단일 관리자 인증만 존재하고 역할 기반 권한 분리, MFA,
    비밀번호 수명주기, 인증 성공·실패 감사 추적이 없다.
  - [RT-01-03] 3 업무 완결성 — 사용자 초대·비활성화, 비밀번호 재설정·복구,
    퇴사자 세션 일괄 철회, 회사별 테넌트 경계가 없다.
  - [RT-01-04] 4 확장성·운영 — 영속 사용자 저장소, 공유 세션 저장소, 운영 모니터링,
    백업·복구와 장애 전환 수단이 없다.
  - [RT-01-05] 5 품질 — HTTP 계층 자동화 테스트는 있으나 실제 브라우저·교차 브라우저,
    부하, 침투 및 외부 보안 분석 검증은 없다.
- 판정: 부분 충족 — 해시 비밀번호, CSRF, HttpOnly 쿠키, 시도 제한과 11개 자동화
  검증으로 단일 서버 로그인 기준은 충족하지만 엔터프라이즈 ERP 인증·운영 기준은 미달한다.
