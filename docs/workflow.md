# 필수 작업 규칙

## 1. 단계별 커밋

- 모든 구현은 의미 있는 최소 단계로 나누고, 단계가 끝날 때마다 반드시 커밋한다.
- 여러 단계를 하나의 커밋에 합치지 않는다.
- 커밋 메시지 형식:
  - 제목: `[단계 N] 한 줄 요약`
  - 본문: 이 단계에서 무엇을, 왜 했는지. 해당 단계를 지시한 사용자 프롬프트의
    요지를 함께 기록한다.

## 2. 회차별 버그 로그 (`docs/log/`)

- 회차(구현 라운드)마다 구현 중 발생한 버그를 `.log` 파일 하나에 기록한다.
- 파일명: `round-NN_<브랜치명>.log` — 브랜치명의 `/`는 `-`로 치환한다.
  예: `feature/inventory` 브랜치의 1회차 → `round-01_feature-inventory.log`
- 버그 하나당 한 블록으로, 시각 / 브랜치 / 증상(에러 메시지 원문) / 원인 /
  수정 방법 / 수정 커밋 해시를 기록한다.
- 형식은 `docs/log/TEMPLATE.log`를 따른다. 버그가 없었던 회차도 파일을 만들고
  "버그 없음"으로 남긴다.

## 3. 충돌 해결 로그 (`docs/conflict-log.md`)

- 작업 중 충돌이 발생하면 — 머지/리베이스 충돌, 의존성 버전 충돌, 스키마·설계
  충돌, 기존 코드와의 비호환 등 — 해결한 뒤 반드시 `docs/conflict-log.md`에
  항목을 추가한다.
- 항목에는 발생 일시, 수행 중이던 작업, 충돌 내용, 원인, 해결 방법(선택지가
  있었다면 왜 그 방법을 골랐는지), 결과를 기록한다. 형식은 파일 상단 템플릿을
  따른다.
- 충돌을 해결한 커밋의 본문에 해당 로그 항목 ID(예: `CONFLICT-003`)를 참조로
  남긴다.

## 4. 지시 → 구현 기록 (`Vibe.md`)

- 사용자 지시(프롬프트)를 받을 때마다 루트의 `Vibe.md`에 항목을 추가한다.
- 항목마다 지시 요지와, 그로 인해 실제 구현된 내용·파일·줄 수를 기록한다.
- 최대한 간단하게: 항목당 지시 1줄 + 구현 1~2줄.

## 5. 회차별 적대적 검증 (`Redteam.md`)

- 매 회차 종료 시 그 회차 구현을 엔터프라이즈 ERP 기준으로 적대적으로 검증하고,
  결과를 루트의 `Redteam.md`에 기록한다.
- **기록 전용**: 검증 결과는 실증 분석 자료로만 쓰며, 이후 코딩 구현에 활용하지
  않는다. 구현은 오직 사용자 지시로만 진행한다.

## 6. 회차별 정량 지표 (`Metrics.md`)

- 매 회차 종료 시 루트의 `Metrics.md` 표에 한 줄을 추가한다: 지시 수, 구현
  줄 수(+/-), 커밋 수, 버그 수, 충돌 수, 레드팀 판정.
- 원천 데이터는 기존 기록(Vibe.md, git diff, docs/log/, docs/conflict-log.md,
  Redteam.md)이며, 기록 전용으로 구현에 개입하지 않는다.

## 7. 이전 설정 무효

- 이전 프로젝트(Centrix, Schedtalk 등)의 프롬프트·스킬·에이전트·규칙은 이
  프로젝트에 적용하지 않는다.
