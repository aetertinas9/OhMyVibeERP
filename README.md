# OhMyVibeERP

중소기업의 업무 흐름을 한곳에서 관리하기 위한 ERP 실증 프로젝트입니다. 로그인과
세션 인증을 기반으로 영업·구매·재고·생산·손익·인사·급여 업무를 연결합니다.

## 실행

Node.js 20 이상에서 별도 패키지 설치 없이 실행할 수 있습니다.

```bash
npm start
```

브라우저에서 <http://127.0.0.1:3000>으로 접속합니다. 개발용 부서별 체험 계정은
다음과 같으며 비밀번호는 모두 `ChangeMe123!`입니다.

| 부서 | 아이디 | 주요 업무 |
|---|---|---|
| 관리 | `admin` | 대표 대시보드와 전체 업무 |
| 영업 | `sales` | 판매처·판매 주문 등록 |
| 구매 | `purchase` | 구매처·품목·발주 등록·안전재고 자동 발주 |
| 생산 | `production` | 품목·재고·BOM·생산 지시 |
| 물류 | `logistics` | 발주 입고·판매 출고·반품·창고 이동·재고 실사 |
| 재무 | `finance` | 월간 매입·판매, 분기 부가세 예상, 입금·지급, 회계기간 마감 |
| 인사 | `hr` | 직원 명부·급여대장·급여명세 |

로그인 후에는 해당 부서에 허용된 메뉴만 표시됩니다. 숨겨진 URL이나 처리 경로에
직접 접근해도 서버가 `403 Forbidden`으로 차단합니다. 구매·영업 담당자는 발주·주문을
등록하고 물류 담당자는 금액이 가려진 문서를 바탕으로 실제 입고·출고만 처리합니다.
관리 계정은 모든 업무에 접근할 수 있습니다.

로그인 후 기준 정보 메뉴에서 다음 데이터를 등록할 수 있습니다.

- 판매처: 우리 회사가 상품·서비스를 판매하는 거래처
- 구매처: 우리 회사가 상품·원재료를 구매하는 거래처
- 품목: 코드, 단위, 과세 구분, 매입·판매 단가, 기초·안전 재고

창고는 `서울`, `인천`, `부산` 세 곳으로 고정되어 있습니다. 품목 등록 시 창고별
기초 재고를 입력하고 `재고 현황` 메뉴에서 품목별 창고 수량과 전체 수량을 확인합니다.
물류 또는 관리 계정은 재고 현황의 `창고 간 재고 이동`에서 출발·도착 창고와 여러 품목을
선택할 수 있습니다. 서울→부산이 기본 경로이며 모든 창고 조합을 지원합니다. 이동 처리 시
모든 품목의 출발 재고를 먼저 검증한 뒤 출발 창고는 차감하고 도착 창고는 같은 수량만큼
증가시켜 한 번에 저장합니다. 한 품목이라도 부족하거나 같은 창고를 선택하면 어떤 재고도
변경하지 않으며 이동 번호·일자·경로·품목·수량·메모가 이력에 남습니다.

월말에 장부와 실제 수량이 다르면 물류 또는 관리 계정으로 `재고 실사·조정`을 엽니다.
창고와 여러 품목의 실사 수량을 입력하면 저장 직전 장부 수량을 고정하고 `실사-장부`
차이를 계산한 뒤, 선택 창고 재고를 실사 수량으로 한 번에 맞춥니다. 0개도 명시적으로
입력할 수 있고 한 품목이라도 잘못되면 어떤 재고도 바뀌지 않습니다. 실사 번호·실사일·
창고·품목별 장부/실사/차이·메모와 조정 처리자·처리 시각이 이력에 남으며 화면에서는
기존 실사 이력을 수정하거나 삭제할 수 없습니다. 마감된 회계월의 실사는 거부됩니다.

`안전재고 자동 발주 제안`은 전체 창고 현재고가 안전재고 미만인 품목을 자동으로 모읍니다.
이미 발주했지만 아직 입고되지 않은 수량을 함께 계산해 `안전재고 - 현재고 - 미입고 발주`
만큼만 추가 발주량으로 권장합니다. 구매 또는 관리 계정은 부족 품목 카드에서 구매처·
입고 창고·예정일·단가를 선택하고 바로 발주할 수 있습니다. 입고 창고는 현재 수량이 가장
적은 창고를 기본으로 제안하지만 담당자가 바꿀 수 있습니다. 발주 버튼을 누르는 순간
권장량을 다시 계산하므로 이미 진행 중인 발주가 부족량을 채우면 중복 추천 발주를 거부합니다.
생성된 문서는 기존 `발주 관리`에 같은 발주 원장으로 남고 이후 물류 입고 흐름을 그대로 탑니다.

`발주 관리`에서는 구매처·입고 창고·여러 품목을 선택해 발주할 수 있습니다. 물건이
도착하면 품목별 입고 수량을 처리하며, 해당 창고 재고가 즉시 자동 증가합니다.

`주문·출고`에서는 판매처의 다품목 주문을 접수합니다. 부분 또는 최종 출고를 처리하면
해당 창고 재고가 자동 차감되고, 실제 출고된 수량 기준의 받을 금액이 누적됩니다.

판매한 물건이 불량으로 돌아오면 원 판매 주문에서 `판매 반품 입고`를 처리합니다. 원출고
창고 재고가 늘고 받을 돈이 원 판매 단가만큼 줄어듭니다. 구매한 물건을 공급처로 보내면
원 발주에서 `구매 반품 출고`를 처리하며 같은 창고 재고와 줄 돈이 함께 줄어듭니다. 이미
입금·지급한 금액보다 반품액이 크면 차액을 고객 환불 예정액 또는 공급처 환급 예정액으로
보존합니다. 출고·입고 수량을 넘는 반품과 재고보다 많은 구매 반품은 거부되며, 반품 번호·
일자·품목·수량·금액·메모가 원 주문에 남습니다. 반품 실행은 물류·관리 계정만 가능합니다.

`받을 돈 · 줄 돈`에서는 실제 출고로 생긴 받을 돈과 실제 입고로 생긴 줄 돈을 판매처·
구매처별로 확인합니다. 재무 또는 관리 계정이 미결 문서에서 입금액이나 지급액을 기록하면
해당 문서와 거래처 잔액이 함께 차감되고 처리일·금액·메모가 정산 원장에 남습니다. 잔액을
초과한 처리와 마감월 이하의 역일자 처리는 거부됩니다. 계좌 잔액을 조회하거나 은행 이체를
실행하는 기능은 아니므로 실제 통장 거래와 대조해야 합니다. 정산 후 반품으로 생긴 고객
환불·공급처 환급 예정액은 표시하지만 실제 환불 지급·환급 입금으로 소멸시키는 기능은 아직
제공하지 않습니다.

`생산 관리`에서는 완제품 1개당 필요한 여러 부품과 수량을 부품 구성표(BOM)로
등록합니다. 생산 지시를 내리면 선택한 창고에서 필요한 부품 재고가 자동 차감되고
완제품 재고가 즉시 증가합니다. 부품이 하나라도 부족하면 어떤 재고도 변경하지 않습니다.

`월간 매입·판매`에서는 조회 월에 실제 입고된 수량의 매입액과 실제 출고된 수량의
판매액에서 같은 달의 구매·판매 반품액을 각각 차감한 금액과 단순 차이를 확인합니다.
거래처·창고·품목별 계산 근거와 원 주문·반품 번호도 함께
표시합니다. 실제 지급·수금과 부가세·급여 등은 반영하지 않으므로 회계상 순이익과는
구분해야 합니다.

재무 또는 관리 계정은 `분기 부가세`에서 한국 시간 기준 분기의 실제 입고·출고와
구매·판매 반품을 과세·영세율·면세로 나눠 봅니다. 저장된 단가는 부가세 별도 공급가액으로
해석하고 과세 품목에는 10%, 영세율·면세 품목에는 0%를 적용합니다. 매출세액에서 공제
가능하다고 가정한 매입세액을 빼 예상 납부세액 또는 환급세액을 표시하며, 거래처·창고·
품목·원문서별 공급가액과 세액 근거도 함께 제공합니다. 일반과세자의 기본 세율과 계산식은
[부가가치세법 제30조](https://www.law.go.kr/LSW/lsSideInfoP.do?docCls=jo&joBrNo=00&joNo=0030&lsiSeq=276117&urlMode=lsScJoRltInfoR)와
[국세청 부가가치세 개요](https://g.nts.go.kr/nts/cm/cntnts/cntntsView.do?cntntsId=7761&mi=2401)를
기준으로 했습니다.

이 값은 신고 편의를 위한 단순 추정치입니다. 세금계산서·카드전표 등 적격 증빙,
불공제 매입세액, 공통매입 안분, 각종 공제·예정고지·가산세·수정신고는 반영하지 않습니다.
또한 품목의 현재 과세 유형을 사용하므로 과거 유형을 변경했다면 원 증빙과 대조해야 합니다.
실제 신고 전에는 [국세청의 공제·불공제 안내](https://www.nts.go.kr/nts/cm/cntnts/cntntsView.do?cntntsId=7802&mi=2355)와
세무 전문가의 검토가 필요하며, OhMyVibeERP가 홈택스 신고서를 제출하지는 않습니다.

재무 또는 관리 계정은 종료된 이전 월을 마감할 수 있습니다. 마감하면 선택 월과 그
이전 월의 발주·판매 주문·생산 지시·급여 확정을 새로 등록하거나 변경할 수 없으며,
현재 월과 미래 월은 마감할 수 없습니다. 마감된 월에 작성된 미결 주문이라도 현재 월에
실제로 처리한 입고·출고는 현재 월 거래로 반영됩니다. 이 화면에는 마감 취소 기능이
없으므로 금액과 거래 근거를 확인한 뒤 실행해야 합니다.

로그인 직후 `대표 대시보드`에서는 전체 창고 합계가 안전재고 이하인 품목, 한국 시간
기준 이번 달 실제 출고 매출, 출고액에서 입금액을 뺀 받을 돈의 현재 합계를 확인합니다.
부족 품목의 서울·인천·부산 수량과 최근 출고·판매처별 받을 금액 근거에서 상세 업무
화면으로 바로 이동할 수 있습니다.

`직원 명부`에는 테스트용 합성 직원 100명이 실제 레코드로 준비되어 있습니다. 직원번호,
이름, 부서·직급, 근무지, 입사일, 이메일과 월 기본급·수당·고정 공제 기준을 검색하고,
새 직원을 추가할 수 있습니다. 합성 데이터는 운영용 개인정보가 아닙니다.

`급여 관리`에서는 귀속월과 월급날을 정해 재직자의 급여대장을 확정합니다. 확정 시점의
기본급·식대·기타 수당·고정 공제를 직원별로 보존하고 전체 또는 개인 급여명세서를
인쇄하거나 PDF로 저장할 수 있습니다. 현재 공제액은 직원 명부의 등록 총액이며 법정
세금·4대 보험 자동 계산이 아니므로 실제 지급 전 세무·노무 담당자의 검토가 필요합니다.

등록 데이터는 기본적으로 `data/master-data.json`에 저장되며 Git에는 포함되지
않습니다. 저장 위치는 `ERP_DATA_FILE` 환경 변수로 바꿀 수 있습니다.

운영 시에는 반드시 환경 변수로 초기 관리 계정을 바꿉니다. 현재 환경 변수 방식은
전체 권한을 가진 관리 계정 한 개를 구성하며, 운영 모드에서는 체험용 부서 계정과
안내가 비활성화되고 세션 쿠키에 `Secure` 속성이 적용됩니다.

```bash
NODE_ENV=production \
ERP_ADMIN_USERNAME=owner \
ERP_ADMIN_PASSWORD='충분히-긴-운영용-비밀번호' \
ERP_ADMIN_NAME='대표자' \
ERP_DATA_FILE='/운영용/절대경로/master-data.json' \
npm start
```

## 검증

```bash
npm test
```

로그인과 보안 경계, 판매처·구매처 분리, 거래처·품목·발주·판매 주문·BOM·생산 검증,
입출고와 생산에 따른 창고 재고·받을 금액·월간 매입판매액 자동 반영, 합성 직원 100명과
급여대장·전체 및 개인 급여명세 생성, 대표 대시보드의 세 핵심 지표, 부서별 서버 권한,
월마감 잠금, 거래처별 채권·채무와 입금·지급 차감, 판매·구매 반품의 재고·금액 역처리와
부서 권한, 다품목 창고 이동의 양쪽 재고 원자 반영, 재고 실사의 장부 차이·원자 조정·
처리 감사·월마감 잠금, 미입고 수량을 반영한 안전재고 추가 권장량과 즉시 발주·중복 방지를
비롯해 실제 입출고·반품 기반 분기 부가세의 과세 유형별 공급가액·매출세액·매입세액·
납부/환급 예상, 한국 시간 분기 경계와 재무 권한을 Node.js 기본 테스트 러너 176건으로
검증합니다.
